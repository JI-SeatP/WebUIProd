# Feature Documentation Format

This document defines the exact structure for documenting features extracted from the old WebUI 
codebase. All docs live in `docs/SOURCE_FEATURES/` and form an Obsidian vault.

## Directory Structure

```
docs/SOURCE_FEATURES/
├── README.md                    ← Vault index with links to all screens
├── S001-WorkOrders.md
├── S002-ProductionEntry.md
├── S003-InventoryCheck.md
├── diagrams/                    ← Draw.io diagrams generated by the diagram skill
│   ├── S001-02-F001-work-order-table-layout.drawio
│   ├── S001-03-F001-quantity-entry-flow.drawio
│   └── ...
└── ...
```

Each screen gets one markdown file. No subdirectories per screen — sections and features are
headings within the file. Diagrams live in a shared `diagrams/` subfolder, named by feature code.

## File Naming

`S{NNN}-{ScreenName}.md`

- `S` prefix + 3-digit zero-padded number
- PascalCase screen name, no spaces
- Example: `S001-WorkOrders.md`

## Document Structure

```markdown
# S001 - Work Orders

> Brief description of what this screen does in the old system.
> Source file(s): `src/old/workorders.cfm`, `src/old/includes/wo_functions.cfm`

## S001-01 Header Section

### S001-01-F001 Company Logo Display
- Displays the company logo in the top-left corner
- Logo is loaded from `/images/logo.png`
- Fixed width: 150px

### S001-01-F002 User Greeting
- Shows "Welcome, {username}" text
- Username pulled from session variable `SESSION.username`
<!-- NOTE: This feature may not be needed in WebUIProd since we have a global navbar -->

## S001-02 Work Order List

### S001-02-F001 Work Order Table
- Displays all open work orders for the current shift
- Columns: WO Number, Product, Qty Required, Qty Produced, Status
- Sorted by WO Number ascending by default
- Data source: stored procedure `sp_GetWorkOrders`

📊 **Diagram:** [Table column layout](diagrams/S001-02-F001-work-order-table-layout.drawio)

### S001-02-F002 Row Selection
- Clicking a row highlights it and loads details below
- Only one row can be selected at a time
- Selected row has a blue background
<!-- NOTE: On touchscreen, rows need to be large enough for finger taps -->

## S001-03 Detail Panel

### S001-03-F001 Quantity Entry Form
- Input field for entering produced quantity
- Numeric keyboard should appear on touch devices
- Submit button saves to database via `saveQuantity.cfm`
- Validation: quantity must be > 0 and <= remaining quantity

📊 **Diagram:** [Quantity entry decision flow](diagrams/S001-03-F001-quantity-entry-flow.drawio)
```

## Coding Scheme

| Level   | Code Format | Example       | Description                          |
|---------|-------------|---------------|--------------------------------------|
| Screen  | `SXXX`      | `S001`        | One per .cfm page/screen             |
| Section | `-XX`       | `S001-01`     | Appended to screen code, one per H2  |
| Feature | `FXXX`      | `S001-01-F001`| Unique within the screen             |

Feature codes (`FXXX`) are globally unique within a screen file. They reset per screen — 
so `S001-01-F001` and `S002-01-F001` are different features.

## Obsidian Wikilinks

Use `[[wikilinks]]` to connect related items:

- Link between screens: `This table is also used in [[S003-InventoryCheck]]`
- Link to TODO items: `Migration tracked in [[MIGRATION_TODO#S001-02-F001]]`
- Link between features: `Depends on [[S001-01-F002|User Greeting]] for session data`

## Inline Comments

Users add observations using HTML comments that won't render in Obsidian preview but are 
visible in edit mode:

```markdown
<!-- NOTE: This is a user observation about the feature -->
<!-- TODO: Clarify whether this feature is still needed -->
<!-- ISSUE: The old code has a bug here where quantities can go negative -->
```

Comment prefixes:
- `NOTE:` — general observation or context
- `TODO:` — something that needs clarification before migration
- `ISSUE:` — known bug or problem in the old code

## Diagram Integration (diagram skill)

When documenting features, **read `ai/skills/diagram/SKILL.md`** and use the diagram skill to
generate `.drawio` diagrams that visually explain feature logic, decision trees, and element
structure. Diagrams make complex features easier to understand at a glance during migration.

### When to Create a Diagram

Not every feature needs a diagram. Create one when the feature involves:

| Feature Characteristic | Diagram Type | Reference |
|---|---|---|
| Branching logic, conditionals, validation rules | **Decision flow** | `diagram/references/decision-flow.md` |
| Multi-step user workflow (submit → validate → save) | **Activity diagram** | `diagram/references/uml-diagrams.md` (Activity) |
| Feature has multiple states (idle, loading, error, success) | **State machine** | `diagram/references/uml-diagrams.md` (State Machine) |
| Complex UI layout with nested elements or zones | **Component diagram** | `diagram/references/uml-diagrams.md` (Component) |
| Data flows between UI, API, and stored procedures | **Sequence diagram** | `diagram/references/uml-diagrams.md` (Sequence) |
| Database tables the feature reads/writes | **ERD** | `diagram/references/database-schema.md` |

**Skip diagrams** for simple, self-explanatory features (e.g., static logo display, text labels).

### Diagram File Naming

`{FeatureCode}-{short-description}.drawio`

- Use the full feature code: `S001-03-F001`
- Append a kebab-case description of what the diagram shows
- Examples:
  - `S001-03-F001-quantity-entry-flow.drawio`
  - `S001-02-F001-work-order-table-layout.drawio`
  - `S002-01-F003-save-production-sequence.drawio`

### Diagram File Location

All diagrams go in `docs/SOURCE_FEATURES/diagrams/`. This keeps the vault clean while making
diagrams easy to find and link.

### Linking Diagrams in Feature Docs

Add a diagram link line directly under the feature's bullet list:

```markdown
### S001-03-F001 Quantity Entry Form
- Input field for entering produced quantity
- Numeric keyboard should appear on touch devices
- Submit button saves to database via `saveQuantity.cfm`
- Validation: quantity must be > 0 and <= remaining quantity

📊 **Diagram:** [Quantity entry decision flow](diagrams/S001-03-F001-quantity-entry-flow.drawio)
```

For features with multiple diagrams (e.g., both a flow and an ERD):

```markdown
📊 **Diagrams:**
- [Submission flow](diagrams/S001-03-F001-quantity-entry-flow.drawio)
- [Data model](diagrams/S001-03-F001-quantity-entry-erd.drawio)
```

### Diagram Style

Always follow `ai/skills/diagram/references/style-guide.md` for the shadcn/Tailwind color tokens.
This ensures all diagrams share a consistent, professional look that matches the WebUIProd UI
design language.

### Workflow Summary

For each feature that meets the criteria above:

1. Document the feature with bullet points as usual
2. **Read** the diagram skill: `ai/skills/diagram/SKILL.md`
3. **Read** the style guide: `ai/skills/diagram/references/style-guide.md`
4. **Read** the relevant diagram-type reference (decision-flow, uml-diagrams, or database-schema)
5. Generate the `.drawio` file and save to `docs/SOURCE_FEATURES/diagrams/`
6. Add the diagram link to the feature documentation
